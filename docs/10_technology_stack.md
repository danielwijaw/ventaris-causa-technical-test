# Technology‑Stack Decision Matrix  

| Technology | Version / Variant | Why it fits the “fast & lightweight” goal | Pros | Cons | Source |
|------------|-------------------|-------------------------------------------|------|------|--------|
| **React** | 18 (concurrent features, hooks) | Mature component model, small runtime (~30 KB gzipped). Supports **React Router v6** for SPA navigation required by the wireframes. | • Rich ecosystem<br>• Declarative UI<br>• Built‑in lazy loading & Suspense for code‑splitting | • Requires a build step (Vite) to stay small | [4] |
| **Vite** | 5.x (native ES‑module dev server) | Extremely fast cold‑start (< 50 ms) and hot‑module replacement, keeping developer feedback loop tight. Produces highly‑optimized bundles via Rollup, helping meet the **< 800 KB gzip** bundle size target. | • Blazing dev server<br>• Out‑of‑the‑box TS support<br>• Minimal config | • Less plug‑in ecosystem than Webpack (rarely an issue) | [9] |
| **Tailwind‑JIT** | Tailwind 3.x (Just‑In‑Time mode) | Generates only the classes actually used, keeping CSS payload ≈ 10 KB gzipped. Aligns with the **Mooovin colour palette** and typography defined in the style guide. | • Atomic utility classes → rapid UI prototyping<br>• Easy theming via config (primary, secondary, etc.) | • Requires build step to purge unused utilities (handled by JIT) | [6] |
| **Zustand** | 4.x (hooks‑based store) | Tiny (< 2 KB) global state manager, perfect for the **cart** and **checkout** state without the boilerplate of Redux. Works seamlessly with React 18 concurrent rendering. | • Simple API (`useStore`)<br>• No Provider nesting | • No built‑in dev‑tools (optional community add‑on) | [4] |
| **MSW (Mock Service Worker)** | 2.x | Intercepts fetch/XHR in the browser, providing the **/api/products** and **/api/checkout** endpoints without a real server. Keeps the project fully static while still exercising real network‑like delays (300 ms search debounce, 1 s checkout). | • No backend needed for the MVP<br>• Works in CI tests | • Requires service‑worker registration (one‑time setup) | [8] |
| **Jest + React Testing Library** | 29.x / 14.x | Fast unit‑test runner; RTL encourages testing the UI from the user’s perspective, fulfilling **NFR‑4** (≥ 80 % coverage). Tests run in Node, no browser needed, keeping CI quick. | • Great DX, watch mode<br>• Integrated mocking | • Limited to unit/integration (e2e would need Cypress) | [2] |
| **GitHub Actions** | Hosted runners (Ubuntu‑latest) | Declarative CI pipeline runs lint, type‑check, tests, and deploys to Vercel/Netlify automatically—meeting **NFR‑5** and **NFR‑6**. | • Free for public repos, easy secrets | • Cold‑start time for each workflow (few seconds) | [2] |
| **ESLint + Prettier** | eslint 8.x / prettier 3.x | Enforces code‑style consistency and catches bugs early, satisfying **NFR‑3**. Both run in the same CI step, adding virtually no extra build time. | • Wide rule set, auto‑fixable<br>• Works with TypeScript strict mode | • Requires configuration upkeep | [2] |

**Bottom line** – This stack delivers a **tiny runtime footprint**, **instant developer feedback**, and **full coverage of the functional & non‑functional requirements** while staying simple enough for a one‑person MVP effort.
